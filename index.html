Chapter 1 什么是架构

对于技术人员来说，“架构”是一个再常见不过的词了。我们会对新员工培训整个系统的架构，参加架构设计评审，学习业界开源系统（例如MySQL和Hadoop）的架构，研究大公司的架构实现（例如微信架构和淘宝架构）……

虽然“架构”这个词很常见，但如果深究一下，“架构”到底是指什么，大部分人就搞不清楚了。例如以下这些问题，你能够准确地回答吗？

要想准确地理解架构的定义，关键就在于把三组容易混淆的概念梳理清楚：

  1. 系统与子系统
  2. 模块与组件
  3. 框架与架构

1.1 系统与子系统

1.1.1 系统和子系统的定义

1.1.1.1 1. 什么是系统？

系统是指由相互关联、相互作用的多个部分（或要素）组成的一个有机整体。这些部分共同协作，完成某种特定的功能或目标。

  • 特性：
    • 整体性：系统的整体功能不是各部分简单功能的叠加，而是各部分协同作用的结果。
    • 结构性：系统由若干个有特定功能的部分或单元组成，这些部分通过一定的组织方式结合在一起。
    • 目标性：系统的存在是为实现某种目标或功能。
    • 动态性：系统内部各部分之间存在相互作用，并随着环境的变化发生动态调整。
    • 开放性：大多数系统都与外部环境进行能量、物质或信息的交换。
  • 实例：
    • 自然系统：生态系统、太阳系等。
    • 技术系统：交通系统、计算机系统等。
    • 社会系统：教育系统、经济系统等。

1.1.1.2 2. 什么是子系统？

子系统是指一个系统的组成部分。换句话说，当一个复杂系统被分解成若干层次或模块时，每一个模块本身又可以看作一个相对独立的系统，这就是子系统。

  • 特性：
    • 相对独立性：子系统可以有自己的结构和功能，可以独立运行。
    • 从属性：子系统是整体系统的一部分，它的存在和功能是为了支持整体系统的目标。
    • 层次性：系统可以有多个子系统，而子系统本身也可能包含更小的子系统。
  • 实例：
    • 计算机系统：操作系统是计算机系统的一个子系统，文件管理又是操作系统的子系统。
    • 汽车系统：汽车系统包括发动机系统、制动系统、电控系统等子系统。
    • 企业系统：一个企业系统可以包含生产系统、营销系统、人力资源系统等子系统。

1.1.2 系统和子系统的关系

  1. 整体与部分的关系：子系统是系统的组成部分，系统是子系统的整体。子系统必须通过协作，支持系统的整体功能。
  2. 层次结构：系统可以被分解为子系统，子系统本身也可以进一步分解成更小的子系统。例如，一个国家是一个系统，省份是它的子系统，而省份的城市又是子系统。
  3. 功能依赖性：子系统的功能是为了支撑系统整体的功能。一个子系统的效率或状态可能直接影响整体系统的运作。
  4. 信息和资源流动：系统和子系统之间通过信息、能量或资源的交换保持相互联系。

1.1.3 举例说明

1.1.3.1 案例 1：交通系统

  • 系统：交通系统
  • 子系统：
    1. 公路系统
    2. 铁路系统
    3. 航空系统
    4. 水运系统

每个子系统内部又可以进一步分解，比如公路系统可以分为高速公路、城市道路和农村公路。

1.1.3.2 案例 2：人体系统

  • 系统：人体
  • 子系统：
    1. 呼吸系统
    2. 循环系统
    3. 消化系统
    4. 神经系统

每个子系统都有其独特的功能，但所有系统协同工作，保证人体的正常运作。

一个系统的架构，只包括顶层这一个层级的架构，而不包括下属子系统层级的架构。所以微信架构，就是指微信系统这个层级的架构。当然，微信的子系统，比如支付系统，也有它自己的架构，同样只包括顶层。

1.2 模块与组件

1.2.1 模块与组件的区别：从抽象到实际

在实际工作中，模块和组件这两个概念经常被混淆。我们常常听到类似以下的说法：

  • MySQL模块主要负责存储数据，而Elasticsearch模块主要负责数据搜索。
  • 我们有安全加密组件和审核组件。
  • App的下载模块使用了第三方的组件。

为什么这些表述中模块与组件的用法容易混淆？根本原因在于，模块和组件的定义并不直观，且两者的边界常常模糊。我们需要从本质上理解它们的区别。

1.2.2 维基百科的定义

1.2.2.1 模块（Module）

模块是一套一致且紧密关联的软件组织，通常包含程序和数据结构两部分。模块作为合成的单位，其接口定义了模块提供的功能以及调用它所需的元素。模块具有独立开发、重用、协作编写和研究的特点。

1.2.2.2 组件（Component）

组件是一个自包含、可编程、可重用、与语言无关的单元。组件可以很容易被组装到应用程序中，是一种强调复用性和物理独立性的概念。

1.2.3 模块与组件的核心区别

如果光看定义，你可能还是感到迷茫。那么我们换一种更直观的解释：模块和组件本质上是系统的组成部分，但它们是从不同的视角拆分系统的结果。

  • 模块是从业务逻辑的角度来拆分系统。它主要体现的是系统的功能划分，其目的在于职责分离。
  • 组件是从物理部署的角度来拆分系统。它主要体现的是系统的运行环境和物理实现，其目的在于单元复用。

1.2.4 形象的理解：模块 vs. 组件

模块更倾向于逻辑上的抽象，而组件则强调实际的物理实现。一个简单的类比是：

  • 模块：像一本书的章节，从内容逻辑上划分功能，每一章负责特定的主题。
  • 组件：像书的装订部分，是实际可触摸的物理构造，如封面、纸张和装订方式。

其实，英文中的 “component” 翻译成中文时，也可以理解为“零件”。零件是物理概念，强调独立性和可替换性，这个特点非常贴合组件的含义。

1.2.5 示例分析：学生信息管理系统

假设我们设计一个学生信息管理系统，这个系统既可以从逻辑层面拆分，也可以从物理层面拆分：

  1. 从逻辑角度（模块）拆分：
    • 登录注册模块：负责用户登录和注册功能。
    • 个人信息模块：负责存储和展示学生的个人信息。
    • 个人成绩模块：负责管理学生的成绩信息。
  2. 从物理角度（组件）拆分：
    • Nginx：作为反向代理服务器。
    • Web服务器：用于处理业务逻辑。
    • MySQL：作为数据存储系统。

模块体现的是功能划分，而组件则是实际实现功能的物理单位。

1.2.6 实际应用：模块和组件如何协同？

如果你是一名业务系统的架构师，在构建系统时，需要同时考虑模块和组件的设计：

  1. 模块的设计：
    • 从业务逻辑的角度出发，明确系统需要哪些功能单元（模块）。
    • 比如，将系统划分为“登录注册模块”“个人信息模块”等，以便于职责分离和逻辑清晰。
  2. 组件的选择：
    • 从物理部署的角度出发，确定实现模块功能所需的技术组件。
    • 比如，选择 MySQL 作为存储组件，但无需关心其内部架构（如 Parser、Optimizer、Caches & Buffers、Storage Engines 等）。

1.2.7 总结

  • 模块强调的是业务逻辑，是从“职责分离”的角度进行系统拆分。
  • 组件强调的是物理实现，是从“复用性和运行环境”的角度进行系统拆分。

在实际工作中，我们需要先从业务逻辑的角度定义模块，再从物理实现的角度选择适合的组件。两者协同工作，共同构建一个高效且可扩展的系统。

1.3 框架与架构

1.3.1 框架与架构：定义与区别

框架和架构是两个非常相似且紧密相关的概念，因此在实际工作中，它们经常被混淆。为了厘清这两个术语的区别，我们可以参考维基百科的定义，并结合实际例子加以说明。

1.3.2 框架（Framework）的定义与特点

框架，通常是指为了实现某种业界标准或完成某些基础任务而提供的软件组件规范，同时也可以指实现这些规范所需的基础功能的具体软件产品。

总结关键点：

  1. 框架是一种组件规范：
    • 比如，MVC（Model-View-Controller）是一个常见的开发规范，类似的还有 MVP、MVVM、J2EE 等。
    • 这些规范为开发者提供了一种标准化的方法去设计和组织代码。
  2. 框架是具体的实现工具：
    • 比如，Spring MVC 是实现 MVC 的开发框架。
    • 除了满足 MVC 规范，Spring 提供了许多额外的基础功能，例如注解（@Controller）、Spring Security、Spring JPA 等，极大地简化了开发流程。

1.3.3 架构（Architecture）的定义与特点

架构指的是软件系统的基础结构，以及构建这些基础结构的原则和规则，并且包括对这些结构的描述。

架构强调的是系统从整体层面的结构化设计，关注的是如何组织系统的各个部分以实现功能和非功能需求（如性能、扩展性）。

1.3.4 框架 vs. 架构

从定义上看，框架和架构的区别相对清晰：

  • 框架关注“规范”：框架定义了一套规则或方法，用于指导开发工作，并提供了一些工具和功能来实现这些规则。
  • 架构关注“结构”：架构描述了系统从更高层次上如何组织，是一种对系统整体设计的抽象视角。

1.3.5 混用与实际说法的误区

尽管定义上有明确的区别，在实际工作中，这两个概念常常被混用。例如：

  • “我们的系统是 MVC 架构。”
  • “我们需要将 Android App 重构为 MVP 架构。”
  • “我们的系统基于 SSH 框架开发。”
  • “这是 SSH 的架构。”
  • “XX 系统是基于 Spring MVC 框架开发的，遵循标准的 MVC 架构。”

这些说法都没有问题！

为什么看似“混淆”的说法其实也能成立呢？关键在于架构的定义中提到的“基础结构”这一概念，它并没有明确说明应该从哪个维度来定义“结构”。我们可以从不同的角度（如逻辑、物理、功能）对系统进行分解，因而“架构”这一概念是多维度的。框架作为一种实现工具，可以成为架构的一个组成部分。

1.3.6 结合示例：模块与组件的延展

以“学生管理系统”为例：

  1. 从业务逻辑的角度看架构：
    • 系统架构可以分为“登录注册模块”“个人信息模块”“成绩管理模块”等，这是功能划分的架构视角。
  2. 从实现工具的角度看框架：
    • 这些模块可能基于 Spring MVC 框架开发，而 Spring MVC 提供了遵循 MVC 规范的工具和基础功能。

从这个角度看，“MVC 架构”描述了系统的逻辑设计方式，而“Spring MVC 框架”描述了实现这种设计的具体工具。因此，说“我们的系统是基于 Spring MVC 框架开发，采用了标准的 MVC 架构”是完全合理的。

1.3.7 总结

  • 框架（Framework） 是一种规范，并通过具体工具为开发者提供功能支持。
  • 架构（Architecture） 是系统的整体结构设计，从多角度描述系统的组织方式。

在实际工作中，框架与架构的界限可能模糊，但理解两者的核心区别有助于更准确地进行系统设计和沟通。

1.4 4R架构

软件架构指软件系统的顶层（Rank）结构，它定义了系统由哪些角色（Role）组成，角色之间的关系（Relation）和运作规则（Rule）。

因为这个定义中的4个关键词，都可以用R开头的英文单词来表示，分别是Rank、Role、Relation和Rule，所以我把定义简称为“4R架构定义”，每个R的详细解释如下。

第一个R，Rank。它是指软件架构是分层的，对应“系统”和“子系统”的分层关系。通常情况下，我们只需要关注某一层的架构，最多展示相邻两层的架构，而不需要把每一层的架构全部糅杂在一起。无论是架构设计还是画架构图，都应该采取“自顶向下，逐步细化”的方式。以微信为例，Rank的含义如下所示：

第二个R，Role。它是指软件系统包含哪些角色，每个角色都会负责系统的一部分功能。架构设计最重要的工作之一就是将系统拆分为多个角色。最常见的微服务拆分其实就是将整体复杂的业务系统按照业务领域的方式，拆分为多个微服务，每个微服务就是系统的一个角色。

第三个R，Relation。它是指软件系统的角色之间的关系，对应到架构图中其实就是连接线，角色之间的关系不能乱连，任何关系最后都需要代码来实现，包括连接方式（HTTP、TCP、UDP和串口等）、数据协议（JSON、XML和二进制等）以及具体的接口等。

第四个R，Rule。它是指软件系统角色之间如何协作来完成系统功能。我们在前面解读什么是“系统”的时候提到过：系统能力不是个体能力之和，而是产生了新的能力。那么这个新能力具体如何完成的呢？具体哪些角色参与了这个新能力呢？这就是Rule所要表达的内容。在架构设计的时候，核心的业务场景都需要设计Rule。

在实际工作中，为了方便理解，Rank、Role和Relation是通过系统架构图来展示的，而Rule是通过系统序列图（System Sequence Diagram）来展示的。
